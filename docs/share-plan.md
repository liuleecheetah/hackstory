# 共享時間軸資料庫規劃（Phase 2 share 層）

**狀態：** S1、S2 已完成；S3 待開發
**最後更新：** 2026-07-21

---

## 0. 核心設計哲學：檔案優先（file-first）

資料庫裡存的不是「一筆一筆的事件」，而是**一份一份完整的 `.hst.json` 檔案**。

- SPEC 第 0 節已把「檔案是發佈與分享的最小單位」定為地基，資料庫設計必須跟隨它。
- 若把事件拆成資料庫的一列一列，SPEC 每次演進資料庫結構就要跟著改；整份存，SPEC 怎麼變資料庫都不用動。
- 使用者隨時能把檔案完整下載走，不被平台綁架。**共用庫關站，時間軸還在。**

> 資料庫的角色是「圖書館」：收藏、編目、讓人找得到。它不是資料的家。

## 2. S1 — 取用（不需登入、不需資料庫）

- 目錄是隨網站部署的靜態檔案 `library/index.json`（放在 `public/library/`），列出精選時間軸的標題、說明、主題、檔案網址。
- 網站上新增「共用庫」面板：列出目錄、一鍵載入成圖層（重用 `adapters/remote.ts` 與現有圖層系統）。
- 也支援貼上任意 `.hst.json` 或 Google 試算表公開網址載入。
- 發佈端暫時用「開 GitHub issue 投稿，由維護者審核後加入目錄」——順便就是內容審核機制。

**架構歸屬：** 目錄的讀取與解析住在 `share/`；面板住在 `ui/`；`core`、`render`、`compose`、`adapters` 一行不改。

## 3. S3 — 發佈（接 Supabase）

Supabase 的適合原因：免費額度大、純靜態網站可直連（不用自己架伺服器）、內建第三方登入。

### 資料表（三張就夠）

| 資料表 | 存什麼 | 白話說明 |
|---|---|---|
| `timelines` | slug、標題、主題、語言、授權、擁有者、最新版號 | 圖書館的「目錄卡片」，用來瀏覽和搜尋 |
| `timeline_revisions` | 完整的 `.hst.json`（jsonb，原封不動）、版號、發佈時間 | 書本身。每發佈一次存一版，舊版不刪——SPEC 的 `meta.revision` 正好派上用場 |
| `profiles` | 使用者名稱、連結 | 作者名片 |

### 規則

- 瀏覽、搜尋、取用：**不用登入**。發佈才要登入（GitHub / Google 一鍵登入，不自己管密碼）。
- 發佈前跑 `core` 驗證器，格式不對不能上架。
- **授權（license）必填**；沒填 `sources` 的事件要提醒（SPEC 第 6 節）。
- 取用後修改再發佈 → 記在 `meta.forkedFrom`（SPEC 第 11 節已預留）。
- **不做即時協作。** 多人同編、衝突合併是 Phase 3 的題目；現階段「各自發佈版本＋fork」就能支撐協作。

## 4. Markdown 作為檔案格式的定位

**結論：Markdown 是「翻譯格式」（住在 `adapters/` 層），`.hst.json` 仍是唯一的正式儲存格式。**

理由：`.hst.json` 的結構——時間精度、相對時間、關係線、軸線、顯示設定——Markdown 沒有標準方式表達。硬要表達就得發明一套嚴格書寫慣例，那是「換一種語法的 JSON」，而且解析器要面對比 CSV 更自由奔放的髒資料。

### 分兩步

1. **匯出成 Markdown 大事記（S2，會做）：** 依時間排序、分軸線、含說明與來源連結的中文大事記，可直接貼進 HackMD、共筆、報導草稿。正中「倡議團體對外說明」「新媒體大事紀」情境。匯出不怕資訊遺失——要完整資料的人拿 `.hst.json`。
2. **從 Markdown 匯入（S4，先不做）：** 若將來要做，可用「Markdown 皮、JSON 骨」——正文給人讀寫，檔尾附一段機器讀的完整 JSON 區塊，兩者對得起來就能無損互轉。但這套慣例值不值得定，要用真實使用者的真實需求檢驗。
